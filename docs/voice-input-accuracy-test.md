# 语音输入准度测试清单 v1.0

> 给银狼桌宠的语音输入做一次系统化准度评估。覆盖你日常会跟银狼说的 10 类场景共 70 句，
> 测完能算出整体字准率（CER），并且能看出**哪一类**最容易翻车 → 直接对应到"加 hotwords 还是
> 升级模型还是改 system prompt"的具体改进动作。
>
> **建议测试用时：约 25–35 分钟**（含填写 + 复盘）。

---

## 0. 测试方法论

### 0.1 环境准备

| 条件 | 标准 |
|---|---|
| 噪声 | 安静室内（< 40 dB），关闭风扇、空调出风口对麦 |
| 麦克风 | 你日常用的那一个（不要特意换"更好的"，要测真实环境） |
| 麦克风距离 | 嘴 30–50 cm，自然坐姿 |
| 语速 | 自然语速（不要刻意慢说"喂模型"，也不要赶飞机） |
| 重复 | **每句只说一次**，不二刷不退磁带 |
| 模式 | 按住 `Ctrl+Alt+V`（hold-to-talk），松手即识别 |
| 网络 | 离线即可（SenseVoice 全本地） |

### 0.2 评分规则（每句三档）

| 标记 | 含义 | 字符级判断 |
|---|---|---|
| **✓ 完全** | 识别结果与预期 100% 一致（标点/末尾"。"可忽略） | CER = 0% |
| **△ 接近** | 1–2 字错（同音错字 / 英文大小写 / 漏标点） | CER ≤ 10% |
| **✗ 错误** | 关键词错、整句不通顺、成词错位 | CER > 10% |

**操作方式**：松手后看 sidebar 入框里的识别结果（agentic 循环还没跑完之前就能看到 user msg），
对照本表"预期"列打勾。如果想严谨点，把"实际识别"列也填上方便事后归因。

### 0.3 给"完全准确"的小宽容

以下情况算 ✓ 完全，不扣分：
- 句末标点 `。` `！` `？` 被吞或加错（SenseVoice 标点能力有限）
- 英文大小写差异（`GitHub` vs `github`）
- 数字数字串 vs 中文数字（`二零二六` vs `2026`）
- 多余/少 1 个空格

### 整体评分

```
准确率 = (✓ × 1.0 + △ × 0.5 + ✗ × 0) / 70 × 100%
```

填写：

- ✓ 总数 = ___
- △ 总数 = ___
- ✗ 总数 = ___
- **加权准确率 = ___% **

### 参考基准

| 准确率 | 评价 | 建议动作 |
|---|---|---|
| ≥ 95% | 优秀 | 现状已经很好，重点优化交互细节 |
| 85–95% | 良好 | 加 hotwords 即可达到 95%+ |
| 70–85% | 一般 | hotwords + 锁 language='zh' + system prompt 容错 |
| 60–70% | 偏低 | 升级到 FP32 模型 + 全套优化 |
| < 60% | 异常 | 排查麦克风 / 采样率 / 模型加载是否成功 |

SenseVoice-Small int8 在 AISHELL-1 (中文) 公测准确率约 96.5%，在你这种"日常+code-switch+专有词"混合场景的合理预期是 **80–88%**（首测，没加优化）。如果首测达不到 80%，先怀疑环境/麦克风，再怀疑模型。

## 3. 失败模式归因

把所有打 △ 或 ✗ 的句子的 "实际识别" 列出来，逐一标记错因（多选）：

| 错因代码 | 含义 | 对应改进项 |
|---|---|---|
| **HW** | 专有词错（"星核猎手" → "心核猎手"） | 加 hotwords 表 |
| **EN** | 英文术语错（"GitHub" → "给特哈布"） | hotwords + 检查 language='auto' 是否合适 |
| **HOM** | 同音字错（"打到极限" → "打倒极限"） | system prompt 加容错 + 上下文 LLM 修复 |
| **NUM** | 数字格式不一致 | 不算硬错，改文档中的"宽容"规则即可 |
| **PUNCT** | 标点缺失/位置错 | 不算硬错，sendMsg 自带触发不依赖 |
| **TRUNC** | 句子被截断（漏字） | 检查 PTT 录音时长（min_duration_ms / 上限） |
| **NOISE** | 环境噪声导致 | 改环境，与模型无关 |

填这个表（每个错句一行）：

| 题号 | 实际识别 | 错因代码 |
|---|---|---|
| __ | | |
| __ | | |
| __ | | |
| __ | | |
| __ | | |
| __ | | |

错因代码的**出现频次**告诉你应该先改什么。比如：
- HW 占主导 → 优先做 hotwords 表（投入最低，收益最高）
- EN 占主导 → 优先 hotwords + 把英文术语加进去
- HOM 占主导 → 优先改 system prompt 让 Claude 容错
- TRUNC 占主导 → 检查 PTT 录音链路而非模型本身

## 4. 改进路线图（基于结果）

按你测出的"加权准确率"和"错因主导项"对照下表挑动作：

| 准确率 | 主导错因 | 第一步动作 | 预期提升 |
|---|---|---|---|
| 80–90% | HW（专有词）| 加 hotwords.txt（10 分钟）| +5–10% |
| 80–90% | EN（英文术语）| hotwords.txt 加英文词 + 锁 `language='auto'`（已是默认）| +3–8% |
| 70–85% | HOM（同音字）| system prompt 加"输入可能来自 ASR"段（5 分钟）| 不改 ASR 但用户感知错误率显著降低 |
| 70–80% | 多种均衡 | 全套优化：hotwords + system prompt + VAD 切静音 | +8–15% |
| < 70% | 任意 | 升级到 FP32 模型（包大 700MB）| +5–15% |

具体改动点见 [docs/voice-input-spec.md](./voice-input-spec.md) 的 v0.2 备选优化章节。
